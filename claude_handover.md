## 🏗️ 1. Architecture Overview (Completed)

The objective was to replace generic, hardcoded ML analytics phrases (e.g., "Accuracy was 40% on Medium questions") with professional, pedagogical feedback generated dynamically by local Ollama (Llama 3.1 8B).

**Data Flow Pipeline:**
`React UI → Node.js `/insights` → Python `/insights` → Structured Signals → NLG Layer → UI`

## ✅ 2. Completed Python Implementation (Do Not Re-Implement)

The Python analytics engine (`analytics-engine/`) has been fully refactored to handle this.

### A. Structured Signal Extraction (`feature_extraction.py`)
A new function `extract_duel_performance_signals(user_id, duel_id=None)` was added. It groups `attempt_history` data for the specific duel and generates structured strings:
```json
{
  "weak_topic": "Quantum Numbers", 
  "strong_topic": "Complex Numbers",
  "speed_pattern": "fast_incorrect", 
  "difficulty_pattern": "easy_fast_medium_slow",
  "consistent_accuracy_topic": "Atomic Structure"
}
```
*Note: This replaces the raw percentages previously sent to the NLG layer.*

### B. Teacher-Style Templates & Fallback (`nlg_formatter.py`)
9 templates were created to act as the gold-standard tone and the deterministic fallback if the LLM fails.
**Template Mapping:**
- `weak_topic` ➔ "Most incorrect responses were associated with {topic}, suggesting this concept may benefit from further review."
- `strong_topic` ➔ "You demonstrated strong confidence in {topic}, answering these questions both quickly and accurately."
- `fast_correct` ➔ "Despite the competitive pace of the duel, you maintained strong accuracy on several questions."
- `fast_incorrect` ➔ "Some responses were submitted very quickly but were incorrect, which may indicate uncertainty with the underlying concept."
- `easy_fast_medium_slow` ➔ "You solved foundational questions efficiently, while more advanced questions required additional time and reasoning."
- `hard_slow` ➔ "Questions requiring multiple reasoning steps took longer to solve, suggesting these problems required deeper analysis."

### C. LLM Prompt Constraints (`nlg_formatter.py`)
The `_llm_generate_teacher_insights(signals)` prompt was updated to enforce:
- Persona: "Educational performance analyst generating professional teacher-style feedback"
- Constraints: Exact 3-4 insights, 15-25 words per sentence.
- Blacklist: "Accuracy dropped", "Low performance".

### D. Fine-Tuning Logger (`nlg_formatter.py`)
A JSONL logger automatically writes every `instruction`, `input` (signals), and `output` (LLM-generated insights) to `analytics-engine/training_data/teacher_feedback_v1.jsonl` for future Llama 3.1 fine-tuning.

### E. Python API Endpoint (`api.py`)
`GET /api/v1/analytics/insights/{user_id}?context=duel_result` now intercepts the request, runs the structured signal extraction specifically for that duel, passes it to the `nlg_formatter`, and returns:
```json
{
  "user_id": "123",
  "context": "duel_result",
  "insights": [
    "Most incorrect responses were associated with Quantum Numbers...",
    "Despite the competitive pace of the duel..."
  ],
  "signals": { "weak_topic": "Quantum Numbers", ... } // For debug
}
```
